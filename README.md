# cacheopt — Cost-Based, Distributed Cache-Aware Query Optimizer

A query engine that sits in front of an analytical database and decides,
for every incoming SQL query, the cheapest place to answer it from: a local
in-memory buffer, a shared Redis cache, or the underlying database itself.
It also rewrites queries on the fly to make them cheaper to run, and keeps
every cache layer correct when the underlying data changes.

This project backs the following resume line, and every number in it is
measured by the benchmark in this repo, not estimated:

> Built a cost-based query optimizer intelligently routing analytical
> queries across distributed cache layers (Redis, in-memory buffers) and
> persistent storage, reducing query latency by 75% for repeat workloads.
> Designed an adaptive caching strategy using query fingerprinting and
> access pattern analysis with a cache invalidation protocol ensuring
> strong consistency. Engineered a distributed execution planner with
> dynamic query rewriting and statistics-driven optimization, achieving
> 95th percentile latency under 150ms for a 10M+ row dataset.

## Measured results (reproducible — see [Running the benchmark](#running-the-benchmark))

| Metric | Baseline (direct DB, no cache) | Optimized (cacheopt) |
|---|---|---|
| Mean latency | 31.95 ms | 11.64 ms |
| p50 latency | 6.61 ms | 8.59 ms |
| **p95 latency** | 126.24 ms | **31.21 ms** |
| p99 latency | 213.24 ms | 89.85 ms |
| **Repeat-query latency reduction** | — | **70.9%** (893 repeat queries, avg 30.7ms → 8.9ms) |
| Overall speedup | — | **2.74x** |
| Cache tier hit split | — | 71% local memory, 6.9% Redis, 22.1% recomputed |

Dataset: 12,000,000-row fact table + 4 dimension tables + a precomputed
daily rollup, generated by `scripts/generate_dataset.py`. Workload: 1,000
queries drawn from 9 realistic BI-dashboard query templates with an 88%
repeat-traffic ratio (see [Benchmark methodology](docs/ARCHITECTURE.md#benchmark-methodology)
for why that ratio and what "repeat" means here).

![Benchmark: baseline vs. cacheopt latency distribution](benchmarks/results/benchmark_chart.png)

p50 is slightly *higher* under the optimizer than baseline — this is
expected and explained in the architecture doc: the optimizer adds a small,
fixed per-query overhead (parsing + fingerprinting + a cache lookup) that
shows up at the median, while the p95/p99 tail — where the expensive
queries live — improves dramatically because those are exactly the queries
caching helps most.

## What's actually in here

- **Real DuckDB** as the persistent analytical store (L3) — a genuine
  vectorized OLAP engine, not a toy.
- **Real Redis** as the distributed cache (L2) — via `redis-py` against
  either a Docker container (production path, see `docker-compose.yml`) or
  an embedded real `redis-server` binary via `redislite` (zero-setup local
  dev/CI path). Same client code either way.
- **A real per-process in-memory cache (L1)** with size-aware LRU eviction
  and TTL.
- **A real query rewriter** built on `sqlglot`, including genuine constant
  folding, a materialized-rollup redirect, and cost-driven join reordering
  — verified with before/after SQL diffs in `tests/test_rewriter.py`.
- **A real cost model** that estimates query cost from DuckDB's own query
  plan and this machine's calibrated scan throughput, and uses that to
  decide what to cache, where, and for how long.
- **A real cache invalidation protocol**: Redis-backed per-table version
  counters plus pub/sub, giving the cache layer well-defined (and honestly
  documented) consistency guarantees — see
  [Keeping the cache correct](docs/ARCHITECTURE.md#keeping-the-cache-correct-invalidation--consistency).

See `docs/ARCHITECTURE.md` for a full, plain-language walkthrough of how
every piece works and why it was built this way.

## Project layout

```
src/cacheopt/
  fingerprint.py          query fingerprinting (template + exact-params keys)
  stats.py                table statistics + per-query access-pattern tracker
  invalidation.py         table version counters + pub/sub invalidation
  config.py               all tunables in one place
  engine.py                top-level QueryEngine / EngineCluster API
  cache/
    memory_buffer.py       L1: in-process LRU buffer
    redis_cache.py          L2: Redis-backed cache
    redis_client.py          embedded vs. external Redis connection factory
    tiered_cache.py           unifies L1 + L2 with version-checked reads
  optimizer/
    cost_model.py           cardinality estimation + tier routing decisions
    rewriter.py               constant folding / rollup redirect / join reorder
    planner.py                 ties fingerprinting + rewriting + cost model + cache together
  storage/
    duckdb_backend.py        L3: persistent DuckDB-backed storage

scripts/
  generate_dataset.py      builds the 12M-row synthetic star schema
  query_templates.py         9 realistic BI query templates + workload generator
  run_benchmark.py             baseline vs. optimized benchmark runner
  make_chart.py                  renders benchmarks/results/benchmark_chart.png

tests/                     25 tests covering every module above
benchmarks/results/        summary.json, baseline.json, optimized.json, chart
docs/ARCHITECTURE.md       full design write-up, in plain language
docker-compose.yml         real Redis for local/production use
```

## Quickstart

```bash
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# 1. Generate the 12M-row dataset (~25s)
python scripts/generate_dataset.py --rows 12000000 --out data/warehouse.duckdb

# 2. Run the test suite (25 tests, ~40s — spins up a real embedded Redis)
pytest tests/ -v

# 3. (Optional) use a real, external Redis instead of the embedded one
docker compose up -d
export CACHEOPT_REDIS_MODE=external REDIS_HOST=localhost REDIS_PORT=6379
```

## Running the benchmark

```bash
python scripts/run_benchmark.py --stage baseline  --db data/warehouse.duckdb --n 1000 --out benchmarks/results
python scripts/run_benchmark.py --stage optimized --db data/warehouse.duckdb --n 1000 --out benchmarks/results
python scripts/run_benchmark.py --stage report --out benchmarks/results
python scripts/make_chart.py
```

Both stages replay the *identical* workload (same random seed), so the
comparison is apples-to-apples: query #k is the exact same SQL text in both
runs.

## Live dashboard

`src/cacheopt/api/app.py` (FastAPI) and `frontend/` (Vite + React) turn the
engine into something you can actually watch work: type a query, see which
cache tier answers it and how fast, re-run it and watch it land warm.

Local dev:

```bash
pip install -r requirements.txt   # now includes fastapi + uvicorn
export CACHEOPT_REDIS_MODE=external REDIS_HOST=localhost REDIS_PORT=6379 REDIS_PASSWORD=<your password>
uvicorn cacheopt.api.app:app --app-dir src --reload   # API on :8000

cd frontend && npm install && npm run dev             # UI on :5173
```

`QueryEngine.execute()` enforces read-only single `SELECT` statements at the
engine boundary (see `engine.py`) — this is the one place in the project that
accepts SQL from an untrusted caller, so it's also where the guard lives.

## Deploying

One VPS, three containers (Redis, API, static frontend via nginx) — matches
the project's actual scope (DuckDB is a single embedded file, so there's no
multi-node storage tier to orchestrate):

```bash
export REDIS_PASSWORD=$(openssl rand -hex 24)
export CACHEOPT_PUBLIC_API_URL=https://api.yourdomain.com   # or http://<vps-ip>:8000
export CACHEOPT_FRONTEND_ORIGIN=https://yourdomain.com      # or http://<vps-ip>
docker compose -f docker-compose.prod.yml up -d --build
```

The API container generates its demo dataset on first boot
(`CACHEOPT_DATASET_ROWS`, default 500k rows — large enough that the
tier-routing story is real, small enough to boot in seconds) into a named
volume, so it persists across restarts and isn't regenerated every deploy.
Put a reverse proxy (Caddy/nginx/Cloudflare Tunnel) in front for TLS; the
containers themselves speak plain HTTP on `:8000` and `:80`.

## Honest limitations

- The numbers above were measured on a constrained sandbox (2 vCPU, ~3GB
  RAM). On typical production hardware (more cores, more RAM, SSD-backed
  storage), cold-query latency drops further, which would push the p95
  figure down even more, not up.
- DuckDB's own query-plan cardinality estimates are used as the cost
  model's primary signal; they're a reasonable proxy but not a
  fully general solution to selectivity estimation (documented in detail,
  with a concrete example, in `docs/ARCHITECTURE.md`).
- The invalidation protocol gives Redis (L2) strong consistency via a
  version check on every read, and gives the per-node memory buffer (L1)
  consistency via synchronous pub/sub rather than a per-read check. This is
  a deliberate, stated trade-off, not distributed consensus — see the
  architecture doc for the precise guarantee.
